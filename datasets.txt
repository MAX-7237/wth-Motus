动作空间公约
自 2026-07-07 本条记录起，ActionFollowingData 后续新训练、数据加载、配置命名和结果记录默认使用 6D rotation / rot6d 动作空间；此前已训练和已评测的 run 默认按历史 Euler XYZ 动作空间理解。后续如需继续使用历史 Euler root，必须在 run name / 配置 / 文档中显式标注 Euler 或 14D。

历史 Euler/source 动作空间为 14D：

[left_dx, left_dy, left_dz, left_droll, left_dpitch, left_dyaw, left_gripper,
 right_dx, right_dy, right_dz, right_droll, right_dpitch, right_dyaw, right_gripper]
当前 Rot6D 动作空间为 20D：

[left_dx, left_dy, left_dz,
 left_r00, left_r10, left_r20, left_r01, left_r11, left_r21,
 left_gripper,
 right_dx, right_dy, right_dz,
 right_r00, right_r10, right_r20, right_r01, right_r11, right_r21,
 right_gripper]
Rot6D 约定：

rot6d = concat(R[:,0], R[:,1])
      = [r00, r10, r20, r01, r11, r21]
其中 R 是 scipy Rotation.as_matrix() 输出的旋转矩阵；历史 droll/dpitch/dyaw 使用 Euler order xyz。训练代码切换到 Rot6D root 时必须显式设置 action_dim=20 / state_dim=20，不得继续沿用旧 14 维配置。

两个当前 Rot6D root 的动作空间语义一致，区别只在容器格式：

root	格式	action/state 存储
/mnt/dataset/csx_workspace/Ideas/data/ActionFollowingData_LeRobot_Rot6D	LeRobot dataset	parquet 中的 action / observation.state 为 20D，并同步更新 meta/info.json、meta/modality.json、meta/stats.json、meta/stats_gr00t.json
/mnt/dataset/csx_workspace/Ideas/data/ActionFollowingBench/data_lerobot_rot6d/robotwin_delta_ee/demo_randomized_zed2i_visible	flat LeRobot task datasets	50 个 task 目录的 parquet 中 action / observation.state 为 20D；非 action/state 资产已物理化为真实文件/目录，root 内无 symlink

Robot frame 坐标系公约
ActionFollowingData / RoboTwin 来源数据的训练用 action 与 state 统一按 robot base frame 解释。该规则覆盖 expert clean、expert randomized 500、enhanced 四类数据，以及 Euler/source 14D root 和 Rot6D/20D 派生 root。

训练用字段包括：

LeRobot parquet 中的 action。
LeRobot parquet 中的 observation.state。
original ActionFollowingData 中的 action.npy。
original ActionFollowingData HDF5 中的 delta_ee_action/vector。
original ActionFollowingData HDF5 中的 delta_ee_state/vector。
当前 aloha-agilex embodiment 下，两臂共享同一个 robot base frame；左臂和右臂的 action/state 不是各自独立的 arm-local frame。robot base frame 轴向约定为：

+X = robot forward / 机器人前方
+Y = robot left    / 机器人左方
+Z = robot up      / 机器人上方
因此，所有 dx/dy/dz 和 state 里的 x/y/z 都按上述 robot base frame 解释。Euler/source 14D 中的 droll/dpitch/dyaw 使用同一 robot base frame 下的 Euler xyz 口径；Rot6D/20D 只把旋转表示从 Euler 换成 rot6d = concat(R[:,0], R[:,1])，不改变坐标系语义。

后续 converter、loader、训练配置和实验记录不得把这些 action/state 解释为 camera frame、wrist frame、per-arm local frame 或其它临时坐标系。若新增非 aloha-agilex embodiment，必须先在本文档中新增对应 robot base frame 轴向说明，再允许作为训练入口。

采样口径
本文只统计 sliding-window 采样，不统计 non-overlapping。

定义：

chunk16 samples = max(L - 16 + 1, 0)
chunk32 samples = max(L - 32 + 1, 0)
训练采样必须是 chunk sample level，不是 family-first 或 dataset-first。

实现规则：

clean / random feasible / counterfactual replay / exploration 是 trajectory-level，按 chunk size 做 stride-1 sliding windows。
perturbed 是 chunk-level，训练时不继续 sliding；每条 33-step sample 只取一个前缀 chunk：chunk16 取前 16，chunk32 取前 32。
必须先展开 effective chunk samples，再按每个 chunk sample 的 family weight 采样。
禁止先按 family 概率选择大类，再在 family 内均匀选择 subtype / task / trajectory / window。
任何训练脚本都必须在 preflight 中打印 effective chunk counts、effective per-chunk weights 和固定 seed sampler audit。
给定当前 manifest 的 effective chunk 数 N_family，真实 sampled family probability 是：

P(family) = N_family * w_family / Σ_i(N_i * w_i)
若目标 sampled chunk count ratio 为 R_family，per-chunk implementation weight 为：

w_family = (R_family / N_family) / (R_ref / N_ref)
本文默认用 perturbed 归一化，因此 w_perturbed = 1。权重与 task set、split、filter、chunk size、perturbed no-sliding 规则绑定，不能跨任务集合无脑复用。
